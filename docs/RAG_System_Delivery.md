# RAG 系统交付文档

## 1. 系统架构简介

- 目标
  - 在现有 Claude Code 框架上实现一个本地向量数据库驱动的检索工具（RAG Tool），用于快速从知识库中检索与查询相关的片段并回传给对话系统进行回答。

- 关键组件
-  RagRetrievalTool
  - 框架内的自定义 Tool，接收参数 query 与 topK，返回 TopK 条相关片段。
  - 当前实现为本地向量数据库的简易演示，后续可对接真实向量数据库（如 Chroma、FAISS、Milvus 等）。
-  本地向量库/知识库
  - 存放待检索的文本片段及元数据的本地存储。初期可使用简单 JSON/文本映射，逐步接入向量化索引。
-  Tool 注册与调用链
  - RagRetrievalTool 与其他内置工具一样，注册到工具系统中，通过 ToolSearch/权限过滤等机制在对话循环中按需调用。
-  API/对接层
  - Rag Tool 的对外接口以 Tool 的输入输出形式暴露给主对话循环，外部可通过 HTTP 网关或直接在测试阶段通过 mock 调用。

- 数据流简述
-  用户提交 query → RagRetrievalTool 接收输入 → 在本地向量库执行检索（TopK）→ 返回结果集（带 id、文本、score 等）给主对话循环 → UI/对话模型渲染结果并展示给用户。

- 性能与容量
-  初始实现以本地演示为主，关注点包括：检索延迟、结果覆盖率、输出长度预算与分页策略。后续可通过向量数据库的索引与并行检索进一步提升。

## 2. API 调用与接口契约

- 概要
- Rag 检索暴露的核心 API 以 Tool 的输入输出模型为基础，便于与对话系统的“tool_use”流程对齐。

- 请求（示例）
- Endpoint（示例，用于内部集成/测试时）
  - POST http://localhost:8000/rag
- 请求体（JSON）
  {
    "query": "请简要解释考研政治中的基础原理与发展趋势。",
    "topK": 3
  }

- 响应（示例）
- 成功
  {
    "results": [
      {"id": "doc-1", "text": "要点内容……", "score": 0.92, "source": "local-vector-store"},
      {"id": "doc-2", "text": "要点内容……", "score": 0.89, "source": "local-vector-store"},
      {"id": "doc-3", "text": "要点内容……", "score": 0.85, "source": "local-vector-store"}
    ]
  }
- 失败/错误
  {
    "error": "description of error",
    "code": 400
  }

- 错误处理
- 400/401/403/500 等错误码，对应客户端应有相应的重试或降级策略。
- 服务端返回空结果时，客户端应有兜底逻辑，避免对话流程中断。

- 性能指标
-  目标检索延迟（单次请求）目标：低于 200-300 ms（本地实现阶段），后续对接向量数据库后再做优化。

- 可执行性建议
-  将测试脚本与文档结合，输出 benchmark_results.csv，以便后续分析并对比版本差异。

## 3. 性能测试（Benchmark）结论

- 目的

- 测试设计
-  场景：5 条典型考研问题的检索性能与稳定性测试。
- 指标（每条查询）：
  - duration_ms：请求端到返回结果的耗时（毫秒）
  - result_count：TopK 的实际返回数量（通常为 3）
  - status：ok / error / http_error
- 测试方法
-  对 Rag Tool 暴露的检索接口进行单次请求，记录耗时与结果数量。
-  采用 benchmark_results.csv 以便后续分析与图表化展示。

-  本地开发环境或 CI/CD 容器，确保在同一环境下重复测试以对比版本差异。

-  Top3 检索准确性：基于人工/半自动评估
-  系统稳定性：在 5 次请求并发场景下无错误
